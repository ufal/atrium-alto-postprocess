A hard-crafted algorithm creates gold data using several models:

- FastText
- Qwen 0.5B

The idea is to train a model on the regression task: score in the range 0.0-1.0

The existing data:
- `Clear` lines - can be messed up to gain `Noisy` and `Trash` variants
- `Noisy` lines - can be auto-corrected to get the semi-`Clear` variants
- `Trash` and `Not-Text` lines - can be used as is, since the score is usually below 0.5
- `Empty` lines- can be ignored

The change of score computed by the algo after auto-correction should be checked

https://ufal.mff.cuni.cz/korektor - example of a possible tool to use (CZ language data)

Maybe apply LLMs like GLM 5.2 to correct the `Noisy` lines

The expected fine-tuned model should be smaller in size than the Qwen 0.5 B used for the perplexity measurements

### Progress update — offline pipeline (Phases 1–4) drafted on `test`

The full **offline** tooling for this issue is implemented under
`tools/quality_model/`, with 75 model-free fast tests (ruff clean). Everything
reuses the **one production scoring engine** (`text_util_langID`) — no second
scorer — so labels stay consistent with the categoriser.

**End-to-end pipeline**

`corrupt.py` → `build_dataset.py` → `correct.py` (+ delta check) →
`train_baseline_gbm.py` + `train.py` → `evaluate.py`

| Phase | What landed |
|-------|-------------|
| **1 – data factory** | `corrupt.py` (12 OCR-noise ops, each aligned to a production detector, SHA-256-seeded), `build_dataset.py` (select → corrupt Clear → **relabel** → dedup → split **by document** with gold docs held out → balance → CSV + manifest), `report_dataset.py` (severity→score monotonicity + synthetic-vs-real realism) |
| **2 – correction** | `correct.py`: korektor **LINDAT REST** (`czech-spellchecker-130202`) + local binary + pluggable **GLM** LLM behind one contract, resumable JSONL cache, Czech→korektor / else→LLM routing. `report_correction_delta.py` answers *"check the score change after auto-correction"* with a **median-Δ go/no-go gate** |
| **3 – models** | `train_baseline_gbm.py` (HistGradientBoosting, trained **± perplexity** — the −perplexity number is the floor the encoder must beat), `train.py` (**`distilbert-base-multilingual-cased`** fine-tune, sigmoid regression head + 3-way category head, Huber + CE), config in `setup/config_quality_model.txt`, run-log in `EXPERIMENTS.md` |
| **4 – evaluation** | `evaluate.py`: vs-algorithm metrics (MAE/RMSE/Spearman, banded + category-head agreement, calibration, predicted-score monotonicity, per-language stratification) **and** the self-reference-free **gold gate** (model-vs-gold F1 vs algorithm-vs-gold F1) |

**Design choices** (full rationale in `agent_dev_logs/plans/23.plan.md`)
- Regress the **raw pre-clamp** score (smooth) + category aux head — category isn't a pure function of the score (per #3).
- **Relabel every variant** with a fresh FastText+Qwen pass (D2), never the stored clamped score.
- Honest about the **self-reference ceiling** (#5): the model can only *distil* the algorithm; "better" is provable **only on the expert gold subset**, which is exactly what the gold gate measures.
- Target model is **< Qwen-0.5B** and removes the perplexity GPU dependency (a ≤5M char-CNN student is the stretch goal that could run CPU-side).

**What's left (needs a real environment, not landed)**
- **Phase 0 production refactor**: extract `build_line_record()` into `langID_classify.py` with a bit-identical parity test — needs FastText+Qwen on a GPU to verify.
- **Real runs**: the model relabelling pass, korektor correction at scale, GBM/DistilBERT training, and the gold-set evaluation (`setup/requirements-finetune.txt` + GPU + the expert gold subset).
- **Phase 5** shadow-mode integration → separate follow-up issue after the gold gate is signed off.

**Open questions before the real runs**
1. Which expert-reviewed subset should be reserved as the untouchable gold test set (the 2,171-line eval corpus, or the 1,348-line review sample)?
2. Confirm korektor **local-install licensing** for cluster-scale correction (REST is fine for the pilot).
