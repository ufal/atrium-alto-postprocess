# Strategy — Issue #2: Text-category definitions & assignment logic

> Issue [#2](https://github.com/ufal/atrium-alto-postprocess/issues/2) · Labels: `enhancement`, `development` · **Status: 🔍 Tocheck**

## Context
#2 is the **definitional root** of the categorisation engine: it fixed the five categories
(Clear / Noisy / Trash / Non-text / Empty) on a *recoverability* axis, the feature-column
contract, and — following the expert review (Dana/Tomáš) and its ChatGPT pattern analysis —
the decision to prefer a **quality-score** over strict rules. On `test` HEAD that decision is
**implemented**: `determine_category(qs, …)` assigns categories from `quality_score` with
`CATEG_TRASH_SCORE_MAX = 0.55` / `CATEG_NOISY_SCORE_MAX = 0.85` as the boundaries, replacing
the earlier cumulative-penalty system documented in the thread. The remaining work is not
*definition* but **boundary calibration** and **gold validation**, which is why #2 is
`Tocheck` and is continued operationally in #3 (calibration) and #5 (importance/ablation).

This plan keeps #2 as the **invariant-owner**: any change to what the categories *mean* or how
`quality_score` maps to them must be reasoned about here first, then calibrated in #3.

## Strategic actions

### A — Lock the definition as a written contract (close the `Tocheck`)
1. Promote the 2026-04-17 column/detector documentation into the repo docs (README or a
   `docs/CATEGORIES.md`) as the **canonical definition**, so the issue thread stops being the
   source of truth. State explicitly that the **quality-score** system is current and the
   **cumulative-penalty** description in the thread is *historical*.
2. Record the two ordering invariants the engine relies on (from #5's findings):
   `CATEG_TRASH_SCORE_MAX < CATEG_NOISY_SCORE_MAX` and `SHORT_PPL_CAP < PERPLEXITY_THRESHOLD_MAX`.
   These are enforced in `validate_constants`; the definition doc should name them so a future
   threshold edit can't silently invert a boundary.

### B — Resolve the two original prompts explicitly
The issue opened with two concrete asks — close them in the doc:
- **digits-attached-to-letters / symbols-inside-words as a separate measure** → satisfied by
  `detect_letter_digit_letter` (`ldl_fuses`) and `detect_strange_symbols` (`symbol`). Mark done.
- **ignore poorly-scored language during assignment** → satisfied by the language-confidence
  gating in the score (low-confidence non-expected langs penalised, not hard-Trashed). Mark
  the decision and its rationale (avoid English-model perplexity false positives).

### C — Hand calibration + validation to #3/#5 (don't duplicate)
- Boundary tuning of `CATEG_TRASH_SCORE_MAX` / `CATEG_NOISY_SCORE_MAX` and the readability/
  low-ppl gates lives in **#3**; #2 only owns whether a *new boundary or category* is needed.
- The "is the current mapping objectively correct" question is blocked on a **gold set**
  (#5/B1, deferred). #2's done-criteria should reference that gold set, not attempt to
  re-derive it.

### D — Future: only if a 6th category or a learned mapping is wanted
The expert review floated a small decision-tree over agreed-label rows. Per #5's close-out,
the right successor is a **monotonic/interpretable classifier** over the same precomputed
features, *once gold exists* (#5/B6). #2 is where that "do we replace the threshold cascade?"
decision belongs — keep it parked here, implemented there.
