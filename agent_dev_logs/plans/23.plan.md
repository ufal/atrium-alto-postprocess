# Strategy — Issue #23: Fine-tuning an NLP model for quality-score assignment

> Issue [#23](https://github.com/ufal/atrium-alto-postprocess/issues/23) · Labels: `enhancement`,
> `development`, `a big one` · Milestone: **Q4 [WP3: Conclusion of Semantic Harmonization]** ·
> **Status: 🟡 PLANNED — strategy + Phase 0/1-core scaffolding drafted; no production code touched
> yet**

## Context

The hand-crafted quality engine assigns each OCR line a score in `[0, 1]` via
`compute_quality_score()` (`text_util_langID.py:1480`) — a normalized weighted sum of **9 signals**
(valid-word `0.35`, weird `0.18`, garbage `0.18`, perplexity `0.08`, vowel `0.07`, lang `0.05`,
gibberish `0.04`, fused `0.03`, length `0.02`, minus a `0.20` short-line penalty) — then routes the
line through ~14 structural rules + QS bands in `determine_category()` (`:1097`) and **clamps** the
stored score into the category band in `categorize_line()` (`:1259`; Trash `< 0.55`, Noisy
`[0.55, 0.80)`, Clear `≥ 0.80`). Two models feed the score: FastText `lid.176.bin` (CPU,
trust-tiered confidence) and **Qwen2.5-0.5B** perplexity on a single dedicated GPU worker
(`langID_classify.py:146`) that up to 32 CPU workers queue into — the pipeline's only GPU
dependency and its throughput bottleneck.

Issue #23 asks to **distill this engine into a single small regression model** (target: score
`0.0–1.0`, size ≪ Qwen 0.5B) trained on algorithm-generated gold, with label diversity manufactured
by:

- **corrupting** `Clear` lines into synthetic `Noisy`/`Trash` variants,
- **auto-correcting** `Noisy` lines into semi-`Clear` variants (korektor for Czech; optionally an
  LLM such as GLM),
- using `Trash`/`Non-text` as-is (score usually `< 0.5`), ignoring `Empty`,
- and **checking the algorithm's score delta after auto-correction**.

Two prior conclusions from the dev logs govern the design:

1. **The self-reference trap (#5, `agent_dev_logs/digests/5.digest.md`).** Labels come from the
   algorithm itself, so training on them can at best *reproduce* the algorithm; it cannot prove
   objective improvement. This issue's honest deliverable is therefore a **cheaper, single-model
   distillation of the engine plus the data machinery** — the corruption/correction continuum is
   what makes the regression learnable off the clamped-band distribution. Any claim of being
   *better* than the algorithm is gated exclusively on the **expert gold subsets** (the 2,171-line
   reviewed eval corpus and the 1,348-line frozen review sample) — the only self-reference-free
   check. Training *on* gold (active learning) stays deferred to a follow-up issue.
2. **Category is not a pure function of the score (#3, `agent_dev_logs/plans/3.plan.md`).**
   Structural overrides flip categories independently of the QS. A score-only model cannot
   reproduce category, so the model gets an **auxiliary category head**, and the structural rules
   stay in production for the integration sketch.

**One-sentence framing:** build a corruption/correction/relabel *data factory* around the existing
scoring engine (reused, never re-implemented — the "one engine, not two" invariant from
`tools/SWEEP_NOTES.md`), train a ≤150 M-parameter multilingual encoder to regress the *raw*
algorithmic score with a category auxiliary head, evaluate against both the algorithm (held-out
docs) and expert gold, and sketch a shadow-mode integration that would retire the Qwen GPU
dependency (FastText stays — its `lang` columns are a pipeline *product*, not just a score input).

## 1 — Training-data construction pipeline

### 1.1 One scoring engine: refactor first (Phase 0)

The blocker to everything else: **the offline replay tool cannot label new text.**
`tools/recategorize_from_csv.py` reuses the *frozen* `perplex` / `orig_lang_score` columns from the
CSVs, so corrupted/corrected variants — new strings — need a fresh FastText+Qwen pass before
`compute_quality_score()` can run. The per-line scoring orchestration currently lives inline in
`process_and_write_batch_cpu()` (`langID_classify.py:315–437`): FastText `remap_lang` + the two-tier
trust system (`TRUST_TIER_TRUSTED` / `TRUST_TIER_UNKNOWN` on the *original* FastText score), the
`SHORT_PPL_CAP` rule (`:333`), garbage/vowel on the **original** (pre-repair) text, then
`compute_quality_score()` → `categorize_line()`. A near-duplicate lives in the service path
`_classify_line()` (`service/text_inference.py`) which **diverges**: it feeds the raw FastText score
(no trust tiers) and skips `SHORT_PPL_CAP`.

**Plan.** Extract a pure function from the batch loop —

```
build_line_record(text_content, original_text, split_ws, split_we,
                  original_lang, original_lang_score, ppl_val,
                  expected_langs, known_lang_bases) -> dict   # every CSV_HEADER field
```

— and make `process_and_write_batch_cpu()` call it. This is a **behavior-preserving refactor gated
on bit-identical CSVs** over the sample corpus (`tests/test_recategorize_parity.py`,
`tests/test_smoke.py` stay green; a new `tests/test_line_record_parity.py` locks the extraction). The
relabeling tool — and later, optionally, the service — then share the one true engine, the same
principle that makes `recategorize_from_csv.py` trustworthy.

> A faithful mirror of `build_line_record()` (calling the **real** production detectors and
> `compute_quality_score`/`categorize_line`, so it is numerically identical, not a second engine) is
> already provided in `tools/quality_model/score_texts.py::build_line_record` as the target of the
> Phase 0 extraction. Phase 0's task is to make production *import* it and to add the parity test.

On top of it, **`tools/quality_model/score_texts.py`** exposes a CLI that loads FastText + Qwen once,
runs `pre_filter_line()` first (recording the route), batches perplexity through
`calculate_perplexity_batch()` (`text_util_langID.py:938`), and emits a full-feature table including
`score_raw` (pre-clamp), `score_clamped`, `categ`, `reason`. It reuses the `gpu_inference_worker`
model-loading conventions (single device, `99999.0` sentinel on failure). Cost bound: the production
classify stage already does `~10⁶` lines per collection run, so a `≤500 k`-line relabel pass is well
under one GPU-night on the cluster's 48 GB card.

### 1.2 Corruption engine: Clear → Noisy/Trash variants

**`tools/quality_model/corrupt.py`** — a registry of parameterized operations, each taking an
intensity `ε ∈ [0, 1]`, composed per line. The taxonomy is deliberately aligned with the detectors so
every engineered signal has a synthetic generator (and the detectors double as unit-test oracles):

| Op                          | Simulates                                                                              | Detector it exercises                                         |
|-----------------------------|---------------------------------------------------------------------------------------|--------------------------------------------------------------|
| `diacritic_strip`           | scanner losing háčky/čárky/umlauts (`á→a`, `č→c`, `ä→a`)                               | `has_cz_diacs`, `compute_vowel_ratio`, FastText confidence   |
| `char_confusion`            | classic OCR confusions: `rn↔m`, `cl↔d`, `l/1/I/|`, `0/O`, `5/S`, `8/B`, `ě→e`, `í→1`   | `score_word` weirdness, perplexity                           |
| `case_flip`                 | mid-word capitals (`sIova`)                                                            | `detect_mid_uppercase`                                       |
| `word_fusion` / `word_split`| dropped / inserted spaces                                                             | `detect_fused_words`, split logic                            |
| `symbol_injection`          | specks / marginalia → `*»~§¤` …                                                       | `compute_garbage_density`, `detect_strange_symbols`          |
| `char_drop` / `char_double` | faint ink / bleed                                                                     | `compute_valid_ratio`, `detect_repeated_chars`               |
| `rotation_ghost`            | inverted / mirrored scan via the `ROTATABLE_CHARS` mapping (`p↔d`, `u↔n`, `b↔q`)       | `compute_rotatable_ratio`, `analyze_rotation_signals`        |
| `ledger_fill`               | table / ledger lines: trailing `TRAILING_FILL_CHARS` runs, digit fragments             | `rule_ledger_fragmentation`, trailing-fill rescue            |
| `vowel_strip`               | vowel-starved scramble                                                                | `rule_vowelless`, `compute_vowel_ratio`                      |
| `truncate`                  | cut lines `≤ 12` chars                                                                 | short-line penalty regime                                    |

**Composition.** For each variant, sample 1–4 ops (weighted) and a severity band; per-op `ε` drawn
within the band. Severity bands are **stratified** so the variants sweep the score range smoothly
rather than clustering. **Determinism:** RNG seeded per variant with
`sha256(global_seed, doc_id, line_num, variant_idx)` → `numpy.random.Generator` (never Python's
salted `hash()`), so the dataset is exactly reproducible from the manifest.

Two built-in validations (Phase 1 report, `report_dataset.py`):

- **Monotonicity** — aggregate Spearman between `ε` and fresh algo score must be strongly negative;
  per-op curves plotted (a flat op is a broken generator).
- **Realism** — feature-distribution comparison (KS distance on `garbage_density`, `word_weird`,
  `vowel_ratio`, `perplex`, `rot_ratio`) between synthetic Noisy/Trash variants and **real**
  Noisy/Trash lines from the corpus CSVs — the direct mitigation for synthetic-vs-real shift.

### 1.3 Auto-correction: Noisy → semi-Clear variants (+ the delta check)

**`tools/quality_model/correct.py`** — a `CorrectionBackend` ABC with:

- `KorektorRestBackend` — LINDAT REST (`https://lindat.mff.cuni.cz/services/korektor/api`,
  spellchecker model), rate-limited with retry/backoff; suitable for the pilot and the gold subsets.
- `KorektorLocalBackend` — subprocess wrapper around a locally installed korektor for cluster-scale
  runs (availability / licensing to verify — see Risks).
- `LlmBackend` — OpenAI-compatible chat endpoint with a fixed "correct the OCR errors, return the
  line verbatim otherwise" prompt (`temperature 0`); satisfies "maybe apply LLMs like GLM", echoing
  the repo's existing GLM precedent (`extract_LLM_ALTO_2_TXT.py`, `glm4v_9b` in `setup/para_config.txt`).
- `NoopBackend` — for tests.

All backends write through a JSONL disk cache keyed by `sha1(backend_id + params + text)` so reruns
are free and interrupted runs resume. Language routing: korektor only for `lang == ces`; deu/eng
Noisy lines go to the LLM backend or are skipped (recorded in the manifest).

**The issue's explicit delta check** — **`tools/quality_model/report_correction_delta.py`**: for
every corrected pair compute `Δ = score_raw(after) − score_raw(before)` via the fresh scoring pass and
report mean/median `Δ`, share improved, share degraded by `> 0.05` (korektor turning garbage into
different garbage), the band-transition matrix (Noisy→Clear / Noisy→Trash rates), per-backend
comparison, and the top/bottom-20 example lines for manual review. **Gate:** a backend with median
`Δ ≤ 0` is unsuitable and its variants are excluded. Individual degraded lines are **kept** with
their fresh labels — the label authority is the scorer, not the assumption that correction helped —
but the report is the sanity instrument the issue asks for, and a natural artifact to share with the
review team.

### 1.4 Relabeling policy

Every dataset row — original Clear/Noisy/Trash lines **and** every synthetic variant — gets its label
from the **same fresh `score_texts.py` pass** (D2): `score_raw` (training target), `score_clamped` +
`categ` + `reason` (aux target and analysis) and the full feature vector (for the GBM baseline and
error analysis). Variants that fail `pre_filter_line` after corruption (e.g. symbol injection pushes a
short line into `Non-text`) are dropped and counted in the manifest (D3).

### 1.5 Dataset assembly

**`tools/quality_model/build_dataset.py`** reads `DOC_LINE_CATEG/*.csv` (cluster corpus;
`data_samples/` + `tests/fixtures/` minis for tests), then:

1. **Select** source lines: exclude `Empty` and pre-filtered `Non-text` (`word_count == 0`); per-doc
   caps so long documents do not dominate.
2. **Generate**: per `Clear` line keep the original + 2–3 corrupted variants at stratified
   severities; per `Noisy` line keep the original + 1 corrected variant; `Trash` lines as-is.
3. **Relabel** everything via `score_texts.py` (GPU stage).
4. **Dedup** on normalized text (strip, collapse whitespace) **before** splitting — repeated
   boilerplate is real in this corpus (that is why `pp_dedup` exists) and is the main leakage vector.
5. **Split by document** (GroupShuffle 80/10/10, frozen seed); the documents of the expert eval
   corpus and the frozen review sample are **excluded from train/val entirely** and reserved for
   Phase 4.
6. **Balance** by soft-capping 20 score bins (report before/after histogram; never upsample by
   duplication across splits).

Schema (parquet): `text`, `score_raw`, `score_clamped`, `categ`, `reason`, `provenance`
(`original | corrupt | corrected`), `ops`, `severity`, `backend`, `source_doc`, `source_line`,
`lang`, `split` + the frozen feature columns. **Size targets:** pilot `≈ 20 k` source lines →
`~60 k` rows (one GPU-evening end-to-end); full `≈ 150 k Clear + 80 k Noisy + 60 k Trash` sources →
`~450 k` train rows after variants and balancing.

## 2 — Model selection

| Candidate                            | Params     | Type            | Verdict                                                                                                                            |
|--------------------------------------|------------|-----------------|-----------------------------------------------------------------------------------------------------------------------------------|
| **distilbert-base-multilingual-cased** | 134 M    | subword encoder | **PRIMARY (D6)** — 3.7× smaller than Qwen, ces/deu/eng covered, fast, Apache-2.0; vocab-trim can later cut ~40–60 % of the size.   |

Head / loss / tokenization: sigmoid-bounded regression head + 3-way category head, `Huber(0.1) +
0.3·CE` (D8); `max_length = 192` subword tokens (lines are `~100` chars, but garbage inflates token
counts 2–3×; log a truncation-rate metric); dynamic padding per batch.

## 3 — Training and evaluation protocol

**Infra.** HF `Trainer` (transformers / torch / accelerate already in `setup/requirements.txt`; add
`pyarrow`, `scikit-learn`, optional `evaluate` in `setup/requirements-finetune.txt`) on one cluster
GPU. Starting hyperparameters: lr `2e-5` (encoder) / `1e-3` (heads), batch 64, 3–5 epochs, 6 % warmup,
weight decay `0.01`, bf16, early stopping on val Spearman, seed 42 (final model re-run with 2 extra
seeds for variance). **Experiment tracking, house style:** each run writes `run_config.json` +
`metrics.json` under a gitignored `runs/`; a committed `tools/quality_model/EXPERIMENTS.md` table (the
`SWEEP_NOTES.md` pattern) records run id, data-manifest hash, model, loss, headline metrics.

**Metrics (`evaluate.py`).**

1. *Vs the algorithm, held-out docs:* MAE, RMSE, Spearman / Pearson on `score_raw`; macro-F1 +
   confusion after banding the predicted score at `0.55`/`0.80` **and** from the category head
   (report both; boundary-focused confusion Noisy↔Clear / Noisy↔Trash as in #3); binned calibration.
2. *Self-reference-free (the only objective gate):* on the expert gold subsets, compare
   **model-vs-gold** category agreement against **algorithm-vs-gold** on the same lines and banding.
   Stratify by language (deu/eng minority) and by provenance.
3. *Structure checks:* per-source-line monotonicity (corruption severity up ⇒ predicted score down,
   pairwise accuracy) and correction ordering (corrected ranked above its uncorrected Noisy source).
4. *Efficiency:* lines/s on GPU and CPU vs the Qwen perplexity path; model size on disk.

**Success criteria (initial targets, revisited after the pilot).**

| Criterion                                    | Target                                                                 |
|----------------------------------------------|------------------------------------------------------------------------|
| Held-out MAE on `score_raw`                  | `≤ 0.06`                                                               |
| Held-out Spearman                            | `≥ 0.90`                                                               |
| Banded 3-way category agreement (held-out)   | `≥ 92 %` overall, macro-F1 `≥ 0.85`                                    |
| Gold-set macro-F1                            | `≥` algorithm's own gold-set macro-F1 `− 0.01` (parity floor)          |
| Monotonicity pairwise accuracy               | `≥ 90 %`                                                               |
| Size / throughput                            | `≤ 150 M` params; `≥ 3×` Qwen-path GPU throughput, or CPU-viable student|
| Beats GBM(−perplexity) baseline              | required — otherwise the encoder is not earning its cost               |

## 4 — Integration path (sketch only — implementation is a follow-up issue)

- **Two-stage shape preserved.** `pre_filter_line()` keeps catching `Empty`/`Non-text` for free; the
  model replaces the expensive middle (Qwen perplexity + the 9-signal scoring) for lines that reach
  scoring. **FastText stays** — `lang`/`lang_score` are pipeline outputs in their own right and it is
  cheap CPU.
- **Plug-in level.** A `QualityModel` wrapper exposed at the `text_util_langID.py` level so both the
  batch pipeline and `service/text_inference.py` inherit it (the same propagation the current
  `compute_quality_score`/`categorize_line` sharing gives). In batch, the existing
  `gpu_inference_worker` (`langID_classify.py:146`) loads the small model instead of Qwen — the queue
  protocol is unchanged, batches just return scores instead of perplexities.
- **Shadow mode first.** A `[QUALITY_MODEL]` config section (`QUALITY_MODEL_PATH`,
  `QUALITY_MODEL_MODE = off|shadow|primary`); in shadow mode the model score is written to a new
  `model_score` column appended to `CSV_HEADER` (`langID_classify.py:105` — safe, the aggregator reads
  by name) while the algorithm stays authoritative; a disagreement report (top-N by `|Δ|`) feeds the
  expert-review loop. Structural rules and document postprocessing stay untouched in shadow mode.
- **Paradata / licensing.** Register the fine-tuned model as a component in `setup/para_config.txt`
  (base Apache-2.0; training-data provenance — korektor-model or GLM-corrected variants — recorded in
  the model card, since `para_licenses.py` ranks CC BY-NC-SA / glm as highly restrictive); the project
  already carries the CC BY-NC FastText floor.

## 6 — Risks and open questions

| Risk / question                     | Assessment                                                                      | Mitigation / disposition                                                                                       |
|-------------------------------------|---------------------------------------------------------------------------------|----------------------------------------------------------------------------------------------------------------|
| **Self-reference ceiling**          | Structural: the model can only match the algorithm except where gold says otherwise | Honest framing (distillation + cost win); mandatory gold gate in Phase 4; gold-*training* punted to a follow-up.|
| **Korektor availability / licence** | REST is rate-limited; local-install licensing unverified (open code, CC BY-NC-SA models) | Verify before Phase 2; LLM backend is the designed fallback; correction is offline, so licence affects provenance docs, not runtime. |
| **Synthetic vs real noise shift**   | Model may learn generator artifacts, not OCR reality                            | Realism report (§1.2); real Noisy/Trash lines kept as-is anchor the real distribution; cap synthetic share `≈ 50 %`; gold eval catches residual shift. |
| **Clamped-score discontinuities**   | Cliffs at `0.55`/`0.80` would poison a regression                              | Solved by D1 (raw target); production comparability restored by banding at eval / integration time.            |
| **Fresh labels ≠ stored CSVs**      | Fresh pass will not bit-match stored clamped scores (clamping, pp flips, drift)  | Deliberate (D2); documented; engine parity is still locked by the Phase 0 refactor tests.                       |
| **GPU availability**                | Relabel pass + training compete with production runs                            | Pilot sized to one GPU-evening; full pass `≈` half a production classify stage; correction stages are CPU/remote.|
| **deu/eng minority coverage**       | Multilingual coverage thin in training data                                     | Language-stratified eval; oversample deu/eng sources if stratified metrics lag.                                 |
| **Subword fragility on garbage**    | Tokenizer explosion could help or hurt                                          | CANINE-s fallback wired into the model registry from the start; truncation-rate metric watched.                 |
| **Document-context signals lost**   | `pp_*` flips and page-context sweeps are invisible to a line model              | Out of scope by design (D2/D3); postprocessing stays in production unchanged.                                   |
